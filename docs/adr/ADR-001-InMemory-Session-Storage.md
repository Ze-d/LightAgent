# ADR-001: 采用内存存储作为会话管理方案

## 状态
Accepted

## 背景

Agent 对话服务需要维护多轮会话上下文。系统需要决定会话历史存储在哪里——是需要持久化到数据库（支持跨进程/跨服务共享），还是仅保存在内存中（单进程内使用）。

**约束条件**：
- 当前系统为单进程部署（uvicorn 单实例）
- 追求快速启动和最小化运维复杂度
- 面向初期阶段，对高可用要求尚未明确

## 备选方案

### 方案 A：内存存储（InMemorySessionManager）
- 会话存储在 Python `dict` 中，随进程存在而存在
- 实现简单，无额外依赖

### 方案 B：Redis 持久化存储
- 会话存储在 Redis，支持跨进程共享
- 进程重启后会话不丢失
- 引入额外运维依赖

### 方案 C：PostgreSQL / MySQL 持久化存储
- 结构化存储会话数据，支持复杂查询
- 引入数据库运维成本

## 决策

采用 **方案 A：内存存储（InMemorySessionManager）**

## 原因

1. **实现简单，无额外依赖**：`dict` 直接可用，不需要安装和运维 Redis/数据库
2. **性能最优**：内存读写，无网络开销和序列化成本
3. **适合初期迭代**：项目尚在早期阶段，过早引入持久化会拖慢开发节奏
4. **会话生命周期与进程一致**：对于单实例部署场景，语义上合理

## 影响

### 收益
- 开发速度快，无数据库依赖
- 零运维成本

### 代价
- **进程重启后会话丢失**：生产环境多实例部署时无法共享会话
- **内存占用随会话数线性增长**：大规模部署有 OOM 风险
- **无持久化保障**：服务崩溃会导致未保存的会话数据丢失

### 技术债
- 若未来需要多实例部署或持久化，需要重构 `BaseSessionManager` 接口并实现新存储后端
- 现有的 `SessionManager` 类（直接使用 dict，无抽象）与 `BaseSessionManager` 接口并存，存在冗余
